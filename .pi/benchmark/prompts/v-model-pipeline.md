# Benchmark prompt — drive the V-Model pipeline from fictive user needs

You are running on a V-Model XT template clone. Read `PI_CONTEXT.md` first
if you have not loaded it yet.

A fictive set of stakeholder needs has been generated by
`.pi/benchmark/scripts/gen-userneeds.py` into:

```
project/01_Requirements/00_UserNeeds/STK-*.md
```

Each STK-*.md already has correct YAML frontmatter per
`.pi/rules/artifact-frontmatter.md`. They are your input. Do not modify them
unless validation forces you to.

## Goal of this run

Drive the V-Model **requirements pipeline** forward end-to-end:

1. **System Requirements (REQ-SYS-*)** — for each stakeholder need (or
   logically-grouped set of needs), derive one or more system requirements.
   Write each as its own file under `project/01_Requirements/02_SystemReqs/`,
   with full frontmatter and at least one entry in `source: [STK-...]`.

2. **Risk Analysis (FMEA-*)** — for each safety-relevant system requirement
   or hazardous-situation pattern visible in the stakeholder set, draft an
   FMEA entry under `project/08_RiskManagement/FMEA-NNN.md`. Reference the
   relevant requirements in `controls:` and `related_requirements:`.

3. **Software Requirements (REQ-SW-AC-*)** — derive software-level
   requirements that realise the system requirements. Use component code
   `AC` for this benchmark unless the stakeholder text strongly suggests
   another component. Place under
   `project/01_Requirements/03_SoftwareReqs/AC/`.

4. **Test cases (TC-*)** — for each baselined requirement, draft at least
   one test case under the appropriate `project/06_Verification/` subfolder
   (unit / integration / system). Use `verifies: [REQ-...]` and, where
   applicable, `mitigates: [FMEA-...]`.

**Stop after step 4.** Do not produce architecture, design, code or any
artifact beyond the requirements + risk + test layer. Those layers are not
in scope for this benchmark.

## Hard rules (validator-enforced)

- Every artifact lives in its own file. Filename = `<id>.md`.
- Use the frontmatter schema from `.pi/rules/artifact-frontmatter.md`. The
  validator will fail any artifact that drops required fields or invents
  vocabulary outside the closed lists.
- Cross-reference by `id` only. No prose links.
- After producing each batch of artifacts, run `make validate` and fix any
  error before moving on.
- A baselined requirement must have `tests: [...]` non-empty. A baselined
  safety requirement must additionally have `risks: [...]` non-empty and
  `safety_class: A|B|C`.

## Quality bar for "good performance" (what the benchmark measures)

The harness will measure your output across these axes. Optimise for them:

- **Coverage.** Every STK ends up referenced as the `source` of at least one
  REQ-SYS. Every REQ-SYS that you mark `baselined` has at least one test.
  Every safety requirement has at least one FMEA.
- **Cleanliness.** `make validate` exits with zero errors at the end of
  the run.
- **G2 readiness.** `make gates` reports `G2_requirements_baselined: READY`
  when you finish — i.e. you have moved enough artifacts to status
  `baselined` to satisfy the gate.
- **Determinism.** This same input may be fed to you multiple times.
  Produce stable artifact IDs and follow consistent derivation rules so
  that two runs on identical input look similar.

## How to start

1. Read `examples/coag-analyzer/` once for the target artifact format.
2. Walk `project/01_Requirements/00_UserNeeds/` and load every STK.
3. Plan a derivation pass (group needs by topic; one REQ-SYS per coherent
   group is usually enough).
4. Produce REQ-SYS files. Run `make validate`. Fix.
5. Produce FMEA files for safety-relevant flows. Run `make validate`. Fix.
6. Produce REQ-SW-AC files. Run `make validate`. Fix.
7. Produce TC files. Run `make validate`. Fix.
8. Finally, `make gates` and confirm G2 READY.

Stop when G2 is READY and validate is clean, or when no further useful
artifact can be derived from the input. Do not pad — empty or duplicate
artifacts will lower the coverage ratio.
