## Medical UI Design Principles

### 1. Safety First — Use Error Prevention

```
HIERARCHY OF USE ERROR CONTROLS (per IEC 62366 / FDA):
┌──────────────────────────────────────────────────────────┐
│ 1. DESIGN OUT (eliminate hazard by design)               │
│    - Physical constraints (keyed connectors, covered     │
│      buttons, hardware interlocks)                       │
│    - Forced functions (can't proceed until confirmed)    │
├──────────────────────────────────────────────────────────┤
│ 2. GUARD (reduce probability or severity)                │
│    - Confirmation dialogs for critical actions           │
│    - Range limits, input validation, format checks       │
│    - Visual/auditory alarms                              │
├──────────────────────────────────────────────────────────┤
│ 3. WARN (inform user of residual risk)                   │
│    - Warning labels, icons, messages                     │
│    - Instructions for Use (IFU)                          │
│    ⚠ WARNING ALONE IS NOT SUFFICIENT for Severity ≥ 8   │
└──────────────────────────────────────────────────────────┘
```

#### Specific Use Error Prevention Patterns

- **Confirmation dialogs**: For START, STOP, DELETE, EJECT, CALIBRATE, RELEASE RESULT
- **Two-step actions**: Press-and-hold for dangerous operations
- **Visual feedback**: Immediate, unambiguous state indication (running, paused, error)
- **Undo capability**: Where clinically safe, allow undo (not for sample processing)
- **Dead-man switch**: For continuous operations that could cause harm
- **Interlock**: UI button disabled until preconditions met (e.g., "Start Run" disabled until QC passed)
- **Progressive disclosure**: Show only relevant options, hide advanced settings behind confirmation

### 2. Clarity — Information Hierarchy

```
VISUAL HIERARCHY FOR LAB DEVICE UI:

┌─────────────────────────────────────────────────────────┐
│ STATUS BAR — Device state, time, user, alerts            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  PRIMARY WORK AREA                                       │
│  - Current task / workflow                               │
│  - Sample information                                    │
│  - Results (most important data)                         │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  SECONDARY INFORMATION                                   │
│  - QC status, reagent levels, maintenance reminders      │
│                                                         │
├─────────────────────────────────────────────────────────┤
│ ACTION BAR — Context-sensitive actions, navigation       │
│ - Left: Back/Cancel  │  Right: Next/Confirm/Start        │
└─────────────────────────────────────────────────────────┘
```

### 3. Consistency Across Components

| Aspect | Rule |
|--------|------|
| **Terminology** | Same term for same concept everywhere (sample ≠ specimen ≠ patient material) |
| **Color coding** | Consistent: Red=Error/Stop, Yellow=Warning, Green=Normal/Go, Blue=Info |
| **Button placement** | Consistent: Cancel left, Confirm right (or platform convention) |
| **Icons** | Standardized across all screens; never rely on icon alone for critical functions |
| **Navigation** | Same navigation pattern throughout: breadcrumb, tab, or wizard |
| **Error messages** | Consistent format: What happened → Why → What to do |

### 4. Error Messages — Medical Grade

```
ERROR MESSAGE TEMPLATE:

┌─────────────────────────────────────────────────────────┐
│ ⚠ [Error Icon]  [Concise Title]                          │
│                                                         │
│ What happened: [Plain language, no technical jargon]     │
│                                                         │
│ Why: [Clinical/operational impact, if any]               │
│                                                         │
│ What to do: [Clear, numbered steps]                      │
│   1. [Action]                                            │
│   2. [Action]                                            │
│                                                         │
│ Error Code: ERR-XXXX  │  Time: HH:MM:SS                  │
│                                                         │
│              [Retry]  [Skip]  [Contact Support]          │
└─────────────────────────────────────────────────────────┘
```

### 5. Result Display — Clinical Grade

- **Results must show**: Value, Unit, Reference Range, Flag (H/L/Abnormal/Critical)
- **Critical values**: Must be visually distinct (red background, flashing until acknowledged)
- **Decimal precision**: Clinically appropriate (glucose: 1 decimal, titer: integer, ratio: 2 decimals)
- **Previous result**: Show for delta check (change from last measurement)
- **QC status**: Always visible alongside patient results
- **Measurement uncertainty**: Display if clinically relevant
- **Interpretation hints**: "Suggests...", "Consistent with...", never "Diagnoses..."

## Multilanguage Support

### Requirements

- **All UI text must be translatable**: No hardcoded strings in source code
- **Unicode (UTF-8)**: Support for Latin, Cyrillic, CJK, Arabic, Hebrew scripts
- **Right-to-Left (RTL)**: Mirror layout for Arabic, Hebrew
- **Locale-aware formatting**: Numbers (1,234.56 vs 1.234,56), dates (DD.MM.YYYY vs MM/DD/YYYY), units
- **Translation memory**: Reuse translations across screens for consistency
- **Regulatory**: IFU translations must match UI terminology exactly

### Implementation Pattern

```cpp
// Framework-independent approach using resource bundles
// All user-visible strings go through translation function:

// ❌ NEVER:
label->setText("Start Run");

// ✅ ALWAYS:
label->setText(tr("ui.run.start"));  // key-based translation
```

### Language Fallback Chain
```
User Preference → System Language → English (default) → Key (error state)
```

### Medical Terminology Considerations
- **Latin terms**: e.g., "in vitro", "ex vivo" — usually kept untranslated
- **Abbreviations**: ALT, AST, CRP — internationally standardized, keep as-is
- **Units**: SI units with locale-appropriate display (mg/dL vs mmol/L)
- **Regulatory terms**: Must match local regulatory terminology exactly

## Accessibility Requirements for Medical Devices

### Why Accessibility Matters in Medical Devices
- Lab technicians may have visual, motor, or cognitive impairments
- Aging workforce in clinical laboratories
- Regulatory trend toward inclusive design (EU Accessibility Act, Section 508)
- **Alternative UI**: Accessibility features serve as alternative interaction modes for all users in suboptimal conditions (gloved hands, bright/dim lighting, fatigue)

### WCAG 2.1 POUR Principles Applied to Medical UI

| Principle | Medical UI Application |
|-----------|----------------------|
| **Perceivable** | High contrast (≥4.5:1 text, ≥3:1 large text), non-color indicators (shapes + colors for flags), screen reader support for all elements |
| **Operable** | Keyboard-only operation (no mouse required), touch targets ≥9mm, no timeouts on critical screens, pause/stop any automated sequence |
| **Understandable** | Plain language (6th-grade reading level where possible), consistent navigation, predictable behavior, input assistance with format hints |
| **Robust** | Compatible with assistive technologies, works across display sizes/resolutions |

### Specific Accessibility Features

| Feature | Implementation |
|---------|---------------|
| **Screen reader** | All UI elements have accessible names/descriptions; results announced with value, unit, flag |
| **Keyboard navigation** | Full tab order, shortcut keys for frequent operations |
| **High contrast mode** | OS-level or application-level high contrast theme |
| **Font scaling** | Support 125%, 150%, 200% without breaking layout |
| **Color-blind safe** | Never rely on color alone; use shape, position, text in addition to color |
| **Audio alerts** | Distinct sounds for different alert types (error, warning, info); volume control |
| **Haptic feedback** | Vibration for critical confirmations (when hardware supports) |
| **Voice control** | Where feasible, support voice commands for hands-free operation |

## UI Testing & Validation

### IEC 62366 Usability Testing

| Phase | Activity | Participants | Output |
|-------|----------|-------------|--------|
| **Formative Test 1** | Paper prototypes, wireframes | 5-8 representative users | Task success rate, time-on-task, confusion points |
| **Formative Test 2** | Interactive prototype | 5-8 representative users | Refined task flows, use error identification |
| **Summative Test** | Final production-equivalent device | ≥15 representative users per user group | Use error rates, root cause analysis, residual risk acceptance |

### Automated UI Testing
- **Unit level**: Test UI logic separate from presentation (MVVM, MVP patterns)
- **Integration level**: Test UI component interaction
- **System level**: Test complete workflows (squish, test automation tools)
- **Accessibility**: Automated WCAG compliance checking
- **Regression**: Screenshot comparison for visual regressions

## Lab Device UI Design Patterns

### 1. Sample Loading / Plate Layout

Visual grid representing physical plate layout:
- Color-coded wells by sample type (patient, QC, calibrator, blank)
- Tap well to assign/identify sample
- Visual feedback for loaded/empty/error wells
- Support for multiple plate formats (96-well, 384-well, strip)

### 2. Run Monitoring

Real-time display during instrument operation:
- Progress bar with estimated time remaining
- Current step description
- Temperature/pressure/optical readings (if relevant)
- Pause/Stop/Emergency Stop buttons (with confirmation)
- Event log (scrolling, filterable)

### 3. Result Review & Validation

Post-run result management:
- Sortable/filterable result table
- Per-result validation (Accept/Reject/Repeat/Rerun)
- QC result overlay (passed/failed per analyte)
- Delta check against previous results
- Batch release workflow

### 4. QC Dashboard

Quality control monitoring:
- Levey-Jennings charts with Westgard rule violations
- QC status per analyte (in-control/out-of-control)
- QC lot tracking and expiration alerts
- Multi-level QC view (Level 1/2/3)

### 5. Maintenance & Diagnostics

Service interface:
- Guided maintenance procedures (step-by-step with images)
- Sensor readouts and diagnostic logs
- Calibration wizards
- Consumable replacement tracking

## Color Palette Guidelines for Medical UI

### Safety-Critical Color Palette

| Color | Hex | Use | Do NOT Use For |
|-------|-----|-----|----------------|
| Red | #D32F2F | Error, Stop, Critical Alert, Abnormal Result | Backgrounds, non-critical info |
| Orange | #F57C00 | Warning, Caution, Out-of-Range | Routine status |
| Green | #388E3C | Normal, Go, Pass, In-Range | Error states |
| Blue | #1976D2 | Information, Help, Navigation | Warnings, errors |
| Grey | #757575 | Disabled, Inactive, Placeholder | Active elements |
| White | #FFFFFF | Background, Cards | — |
| Near-Black | #212121 | Primary Text | Backgrounds |

### Color-Blind Safe Alternatives
- **Pattern + Color**: Striped warning bars, not just color change
- **Shape + Color**: Circle(●)=Normal, Triangle(▲)=Warning, Octagon(⏸)=Stop
- **Position + Color**: Left=Back/Cancel, Right=Confirm — position is color-independent

## Font & Typography for Medical Displays

| Level | Size | Weight | Use |
|-------|------|--------|-----|
| H1 | 24-28px | Bold | Screen title |
| H2 | 18-22px | Semibold | Section header |
| Body | 14-16px | Regular | Content, instructions |
| Results | 20-24px | Bold | Patient results (must be highly legible) |
| Units | 12-14px | Regular | Alongside results |
| Caption | 11-12px | Regular | Timestamps, metadata, error codes |

**Font Requirements:**
- Sans-serif (Roboto, Inter, Noto Sans for multilanguage)
- Monospace for error codes, serial numbers, barcodes
- Monospaced digits (tabular figures) for numeric columns in tables
- Minimum 14px for critical information (results, alerts)
- Line height: 1.4–1.6 for body text, 1.2 for headings
