# ADR-002: Hardware Safety Interlocks

## Decision
Safety-critical interlocks (lid-closed, emergency stop) shall be implemented in hardware as the primary safety path, with software monitoring as a secondary/diagnostic path.

## Context
Risk analysis identified:
- **FMEA-004**: Biohazard exposure (severity 9) — lid must prevent operation when open
- **FMEA-005**: Mechanical injury (severity 5) — moving parts must stop if accessed
- **FMEA-007**: Cannot stop (severity 7) — emergency stop must work even if software frozen

The risk control hierarchy (ISO 14971) mandates:
1. Inherent safety by design > 2. Protective measures > 3. Information for safety

A hardware-based interlock is level 1 (design) — it eliminates the hazard by preventing motor operation when unsafe. A software-only interlock would be level 2 at best.

## Implementation

```
Motor Power Supply ──┬── NC Emergency Stop ──┬── NC Lid Switch ──┬── Motor Driver Enable
                     │                       │                   │
                     └── (bypass for test?)  └── (bypass?)       └── MCU GPIO (monitor)
```

The motor enable signal is the logical AND of:
- Emergency stop NOT pressed (NC contact closed)
- Lid fully closed (NC contact closed)
- MCU software enable (GPIO high)

All three must be true for the motor to run. Any one going false cuts motor power immediately — no software involved.

## Consequences
- **Positive**: Motor cannot run with lid open even if MCU crashes, firmware has bug, or watchdog fails
- **Positive**: Easier safety case for regulatory review (hardware interlock is deterministic)
- **Negative**: NC switches and wiring add cost and failure modes (broken wire = motor won't run, not unsafe)
- **Negative**: Cannot implement "service mode" to run motor with lid open without a physical bypass (intentional — prevents misuse)
