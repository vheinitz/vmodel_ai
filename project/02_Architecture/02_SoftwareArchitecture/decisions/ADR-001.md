# ADR-001: Single-MCU Embedded Architecture

## Decision
The device shall use a single microcontroller (MCU) as the sole computing element, with no separate PC, single-board computer, or touchscreen HMI.

## Context
The liquid mixer requires:
- Reading 2–3 digital inputs (buttons, lid interlock, emergency stop)
- Driving 1 DC motor (PWM + direction + enable)
- Driving 3 LEDs and 1 buzzer
- Running a simple state machine (idle → mixing → done/error)
- No display, no networking, no database, no user accounts

This is well within the capability of a single mid-range MCU (e.g., STM32G0, ATmega328P).

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Single MCU** (chosen) | Lowest cost, fastest boot, simplest regulatory path, no OS to qualify | Limited expandability, harder to add GUI later |
| MCU + Raspberry Pi | Touchscreen possible, Linux ecosystem, networking built-in | 10× cost, 30+ s boot, SOUP qualification (Linux kernel), SD card reliability concerns |
| MCU + ESP32 | WiFi/BT for future LIS, still low cost | Adds wireless SOUP (security concerns), overkill for current requirements |

## Consequences
- **Positive**: Minimal BOM, simple firmware, no OS qualification, fast boot (< 1 second)
- **Negative**: If requirements later expand to include touchscreen, networking, or LIS connectivity, a hardware revision would be needed
- **Mitigation**: Reserve UART/I²C/SPI headers on the PCB for future expansion module attachment
